# dev-prep-checklist v2 — コーディング開始前チェック

## Step 0: 新規/継続の判定(まずこれ)

「このタスクは新規プロジェクト or 未経験技術か?」

- **YES(新規/未経験)** → フルチェック(Step 1〜5)
- **NO(既存プロジェクトの継続)** → 簡易チェックへ

## 簡易チェック(既存プロジェクト継続時)

1. STATUS.md / PLAN.md を確認したか(最新の進捗・既知の問題)
2. 対象プロジェクトとCCインスタンスは一致しているか(env-guard と連携)
3. SemVer は前回と一貫しているか
→ OKならそのまま実装指示へ直行。フルチェックは行わない。

## フルチェック(新規開発時)

1. **タスク明確化**: 何を作るのか。完了条件は何か。曖昧なら質問する。
2. **既存パターンの流用確認**: 確立済みパターンを新規発明より優先する。
   - 単一ファイル構成 + config.js の APP_VERSION 単一定数
   - i18n は i18n.js 方式(ja/en)
   - 引き継ぎは STATUS.md / PLAN.md
   - 認証は下記「認証方針」参照
3. **過去会話・過去資産の確認**: 同じ話題・同じ機能を過去に
   作っていないか確認する。
4. **環境の特定**: どのプロジェクト・どのCCインスタンスでの作業か明示する。
5. **技術的事前確認**: 不明点は推測せず質問する。勝手に進めない。

## 認証方針(2026-06改訂)

- **デフォルトは認証なし。** 管理機能が必要になった時点で、
  何を・誰から守るのか(脅威モデル)を定義してから方式を選定する。
  - UI操作を隠したいだけ → 管理コードを公開サイトに含めない(非公開化)
  - データを守りたい → データ自体の暗号化(Web Crypto / AES-GCM)
  - 本物のアクセス制御 → サーバーまたはエッジが必須
    (ホスティングロックインとのトレードオフに注意)
- パスワード・ユーザー名・APIキー・ハッシュ値をソースに
  ハードコードしない。保存方法は実装前に必ず確認する。
- 認証情報をCLI引数で渡さない。

## 出力形式

チェック結果を以下の形式でまとめ、OKを得てから実装に進む:

```
【タスク】●●
【判定】新規 / 継続
【流用パターン】●●
【影響ファイル】path/to/file
【コミットメッセージ案】●●(日本語)
```

## 重要原則

1. コードを書き始める前に必ずチェックリストを通す。飛ばさない。
2. 各ステップで何をしているかを分かりやすく説明する。
3. 不明点は推測せず質問する。
4. 既存パターンの再利用を新規発明より優先する。
5. マルチCC環境では環境とCCインスタンスを必ず明示する。

## 発動しないケース

- 単なる質問、ドキュメント閲覧、雑談
- 既にチェックリストを通過したタスクの継続作業
